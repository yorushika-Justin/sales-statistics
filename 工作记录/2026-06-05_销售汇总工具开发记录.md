## 九、经验沉淀

1. **xlrd 1.2.0 是 xls 读取的最后一个支持版本**，2.0+ 已不支持；装新版前要明确数据格式

6. **tkinter 弹窗** 是单文件 exe 内置错误提示的最简方案，零依赖

## 十一、v1.3 功能更新：日期来源修复 + 图片宽度自适应

**Bug 1 — 日期来源错误**: 文件名反映的是供应商系统导出时间（如 `_2026-06-05_215846` = 6/5 21:58:46 导出），而非实际数据日期。xls 内部 Row 2 明确标注 `日期:2026-06-03 00:00:00 至 2026-06-03 23:59:59`。工具原用 `extract_date_from_filename()` 取文件名日期，导致 PNG 标题和文件名与数据不匹配。

**修复**:
- 新增 `_extract_date_from_xls(sh)` 函数，从 xls sheet 的 Row 2 提取日期范围
  - 起止同一天 → `2026-06-03`
  - 跨多天 → `2026-06-03至2026-06-05`
  - 提取失败 → fallback 到文件名日期
- `read_xls()` 返回值改为 `(data, date_str)` 元组
- `main()` 和 `on_drop()` 改用 `read_xls()` 返回的日期

**Bug 2 — 图片宽度固定导致排版溢出**: 图片宽度固定 560px，多日长日期标题（如「销售汇总  2026-06-03至2026-06-05」）在 32px 字号下远超 560px，标题溢出图片边界。

**修复**:
- `draw_table()` 先计算标题宽度和最宽品牌名宽度
- 图片宽度 = `max(品牌列宽 + 金额列宽, 标题宽) + 2×边距`
- 品牌列宽 = `max(260, 最宽品牌名宽 + 40)`

**验证**: 多日文件测试通过，日期正确提取为 `2026-06-03至2026-06-05`，图片宽度自适应为 598px，标题完整显示。

### 12.1 需求来源

2. **日期逻辑修正** — 之前 PNG 标题用的日期可能比数据实际统计日多 1 天

### 12.2 关键概念：数据滞后性

xls 的 Row 2 实际是**导出时刻**看到的时间范围：

| Row 2 范围 | 实际含义 | 实际最后完整统计日 |
|---|---|---|
| `2026-06-05 00:00 至 2026-06-05 23:59` | 当天导出当天数据 | 2026-06-05 |
| `2026-06-05 00:00 至 2026-06-06 23:59` | 6.6 当天导出 6.5 整天 + 6.6 部分 | 2026-06-05 |
| `2026-06-05 00:00 至 2026-06-07 23:59` | 6.7 导出 6.5+6.6 整天 + 6.7 部分 | 2026-06-06 |

> **核心规则**: xls 的"结束日期"代表截至 23:59:59 那一刻，但**当天数据通常未完整**（因为是提前导出的）。所以"最后完整统计日" = `end_date - 1 天`。
>
> 例外：当 `start == end` 时（当天导出当天），起止同日就是完整日，显示 `start` 即可。

### 12.8 v1.4 日期边界 bug（**待修**）

**Bug 描述**（在第 25 轮审计中发现）：

当前 `_extract_date_from_xls()` 的核心逻辑：

```python
end_date = datetime.strptime(end_str, '%Y-%m-%d')
actual_end = end_date - timedelta(days=1)  # 总是减 1 天
actual_end_str = actual_end.strftime('%Y-%m-%d')
if start_str == actual_end_str:
    return start_str
return f'{start_str}至{actual_end_str}'
```

**问题**：当 `start == end`（当天导出当天数据）时，会无脑 `end - 1`，导致 `start` 变成 `end-1` 的下一天，输出形如 `2026-06-05至2026-06-04`（顺序倒置），应该输出 `2026-06-05`。

**复现验证**（`python -c` 跑 `_extract_date_from_xls` 模拟）：

| 输入 (start, end) | 实际输出 | 期望输出 | 是否正确 |
|---|---|---|---|
| `2026-06-05, 2026-06-05` | `2026-06-05至2026-06-04` | `2026-06-05` | ❌ |
| `2026-06-05, 2026-06-06` | `2026-06-05` | `2026-06-05` | ✅ |
| `2026-06-05, 2026-06-07` | `2026-06-05至2026-06-06` | `2026-06-05至2026-06-06` | ✅ |

**触发条件**：用户当天导出当天数据（例如下午 3 点查看当天销售并导出）。

**修复建议**（伪代码）：

```python
if start_str == end_str:
    return start_str  # 当天导出当天, 无滞后, 直接返回
end_date = datetime.strptime(end_str, '%Y-%m-%d')
actual_end = end_date - timedelta(days=1)
actual_end_str = actual_end.strftime('%Y-%m-%d')
if start_str == actual_end_str:
    return start_str
return f'{start_str}至{actual_end_str}'
```

**影响范围**：当前测试数据 `供应商日销售汇总_2026-06-06_152104.xls`（start=06-05, end=06-06）未触发该 bug，但 `技术文档.md 4.1` 描述「单日→`YYYY-MM-DD`」的承诺与实际代码不符，需同步修正。

**修复状态**：✅ **v1.5 已修复**（2026-06-06）。在 `_extract_date_from_xls()` 中 `end - 1` 之前增加 `start == end` 提前返回判断。

### 15.4 v1.1.1 hotfix

**Bug 描述**：双击 `启动网页端.vbs` 后浏览器没自动打开。

**诊断**（systematic-debugging Phase 1-2）：
- .bat 启动（python.exe）→ 浏览器**能**开
- .vbs 启动（pythonw.exe）→ 浏览器**不**开
- 手动开 `http://127.0.0.1:5000` → 能开
- **根因**：`webbrowser.open()` 在 `pythonw.exe`（无控制台）下不工作（Python 社区已知问题）。`python.exe` 下正常。

**修复**：

```python
# 改前
webbrowser.open('http://127.0.0.1:5000')

# 改后
os.startfile('http://127.0.0.1:5000')
```

`os.startfile` 是 Windows Shell API 调用，pythonw 下稳定。同时**删除**了原 `open_browser()` 里的 `urlopen` 探测逻辑——该逻辑在首次启动时也会跳过打开浏览器（Flask 启动快于 1 秒 → urlopen 探测成功 → 误判"已有浏览器" → 不开）。二次启动场景由 `_check_already_running()` 处理，`open_browser()` 不再需要"避免重复开"判断。

**改动**：`web_server.py` 的 `open_browser()` 函数 11 行。

**验证**：用户重测 .vbs 启动 → 浏览器自动打开 ✅

### 15.5 经验沉淀

1. **pythonw + `webbrowser.open()` 兼容性问题** — Windows + Python 已知坑。`os.startfile(url)` 是稳定替代（Windows Shell API，不依赖 Python 的 webbrowser 模块）。
2. **冒烟测试不替代端到端** — subagent 用 `Invoke-WebRequest` 抓 HTML 验证 ≠ "浏览器自动打开"。前者会通过，后者首次会失败。**带 UI 交互的功能必须用户实测**。
3. **subagent-driven 对 tightly coupled task 不适合** — Task 3-8 都改 `web_server.py`，串行 6 个 subagent 浪费 context（每个 subagent 都要重新读 350+ 行文件）。合并为 1 implementer + 综合 reviewer 更经济（节省 ~15 个 round trip）。
4. **YAGNI 优先 + 用户表态** — code quality reviewer 频繁提"小改进"（双 close 改 with、TclError 兜底、魔法数字提取等），用户明确表态"就用你自己写的 spec" → 不修，保持 spec 一致性。
5. **urlopen 探测"避免重复开"是反模式** — urlopen 是主动 HTTP GET，无法判断"是否有浏览器在访问"。原注释"服务已响应，可能已有浏览器打开"是错误假设。

## 十七、品名汇总功能设计讨论（2026-06-08）

**状态**: 设计阶段，待用户审阅后实施

### 17.1 需求来源

用户在 Edge 浏览器用网页端方案从 `工作记录/供应商日销售汇总_2026-06-08_111355.xls` 生成品牌汇总 PNG（参考 Image #1），希望**额外生成"展示分类"的 PNG**（参考 Image #2 — 供应商系统原始明细表）。

## 二十五、v1.7 新增 xlsx 格式支持(2026-06-08)

### 25.1 需求

用户询问 xls 和 xlsx 的区别,确认需要支持 xlsx 格式。

### 25.2 技术方案

| 格式 | 库 | API 差异 |
|---|---|---|
| `.xls` | xlrd 1.2.0 | 0-indexed, `sh.cell_value(r, c)`, `sh.nrows/ncols` |
| `.xlsx` | openpyxl 3.1.5 | 1-indexed, `sh.cell(r, c).value`, `sh.max_row/max_column` |

**关键差异**:
- xlrd 行列 0-indexed, openpyxl 1-indexed
- xlrd Row 2 = openpyxl Row 3
- xlrd `wb.sheet_by_index(0)` = openpyxl `wb.active`
- openpyxl 需要 `read_only=True, data_only=True` 优化性能

### 25.3 代码改动

| 文件 | 改动 |
|---|---|
| `汇总脚本.py` | 新增 `import openpyxl`；新增 `_extract_date_from_xlsx(sh)`；新增 `read_xlsx()`；新增 `read_xlsx_items()`；`on_drop()` / `main()` 根据后缀分发；GUI 提示更新 |
| `网页端/web_server.py` | import 加 `read_xlsx` / `read_xlsx_items`；`/upload` 路由根据后缀分发；前端 accept 改为 `.xls,.xlsx`；提示文字更新 |
